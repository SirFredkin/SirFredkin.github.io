软件的定义

软件的定义

说白了，就是指令+数据结构+文档+服务

指令是可以执行的，执行之后就能帮我们实现功能。

数据结构：允许程序能够合适地去访问、修改数据。

文档描述了程序能做什么。

服务是针对用户的支持，让他们更好的应用软件。

还有一点比较重要，就是软件的双重作用：”它既是产品，又是交付产品的载体。“这句话怎么理解？有编程经验的人都知道，程序会运行一些指令，将数据进行处理，这是它能实现的功能，这时软件作为产品。而操作系统、网络，我们知道这些应用的实现需要软件、指令。这时软件作为产品交付的载体。

软件的特点

最开始，我们需要明确这样一件事：软件是一个逻辑产品。软件有这样几个特点，这些应该很好理解：

a.软件是被开发出来的，不是像零件一样被制造。

b.软件会退化。这一点体现为软件发生错误的概率会随着一次次改动而不断增加，下面的磨损曲线应该能给你一个不错的理解。每一次更改都可能引入新的错误，错误率通过后续的维护来降下来。

c.Custom-built，我们知道在机械行业有零件，用零件来组成一个机器。但软件行业没有那么多规范化的“部件”，虽然现在的开发也有向那方面发展的趋势。

d.复杂

软件危机和软件工程

IBM开发的360机的操作系统，是软件危机的一个经典案例。问题主要在于之前的软件开发方式，因为工作量小，所以只需要自己开发就可以，倒也不会有什么问题。不过IBM360机的操作系统写出了100万行源代码，有1000人投入开发工作。工作量大，就会有协作等各方面的问题出现。

软件危机的一些体现：

比如说，开发遇到问题的时候，预算会严重超支，时间也可能会拖延。

而软件工程最初的目的，就是为了解决软件危机的。

软件工程实际上不是告诉我们编程中的规范，而是告诉了我们方法学的东西。具体来讲，就是：将系统的、规范的、可量化的方法应用于软件的开发、运行和维护，将工程化的方法应用于软件。

所以我们可以这么说：

软件工程=计算机科学+管理工程

我对于“工程”的理解，可能有些偏差：

工程不像基础科学、理论，是要来解决实际问题的，是有预算的，比如我们要做一个数值模拟，我们可以选择增加精度，但代价是开发的难度、时间和运算的速度，那么我们需要做出取舍(trade-off)，选个能满足要求的，整体而言代价又最低的就好。

软件工程也是一样，比起强调质量，在性能和交互上做到尽善尽美，但如果因此大幅增加了开发成本、开发时间的增加、影响了后续的开发步骤的话，也会带来问题。

2.软件过程

什么是软件过程？它大体上告诉了我们在开发软件的时候需要完成的活动、行为和任务集，也指定了相关的要求。每一个过程由许多更小的活动组成，它们被称为框架活动。在即将讲到的通用过程模型中，定义了五个框架活动，它们无论什么规模的软件都需要被采用，不过它们执行的程度和严格程度是根据比如项目的种类和特点、团队的特点而有不同的。这五个活动是：

交流：
主要是和共利益者(stakeholders，比如说客户、最终用户、软件销售、开发经理)交流。

计划：

建模：
这里又要建两个模型，一个是需求模型，另一个是设计模型

构建：
更细致的话也分两步：生成代码 和 测试

部署：

除去框架活动，软件工程也定义了一些Umbrella Activity，这些活动在整个软件的开发过程中都被采用，它的作用是帮助开发团队控制进度、质量、改变和风险。说白了就是保障开发能顺利进行的保护活动。举几个例子：风险管理、项目管理和跟踪、项目控制。

通用的过程模型 Generic Process Model

通用过程定义了五个框架活动，主要目的就是力图为开发带来秩序。也会有其他不同的过程模型存在。它们也要完成那五个活动。过程模型定义的，是这些活动之间的 完成先后顺序 和 它们之间的关系。

过程流 Process Flow

在介绍具体的一些过程模型之前，最好先介绍一下Process Flow的概念。过程流就是刚刚说的活动的执行先后顺序。有以下四个执行顺序：线性的(Linear)、迭代的(Iterative)、演化的(Evolutionary)、和并行的(Parallel).

瀑布模型 Waterfall Model

线性的执行下去。

优点：是最广泛的一种、是一个可以被采用的模版。

缺点：

1.无法随客户需求改变而改变。

2.在开发到最后才能得到一个可执行程序给客户看。

3.开发人员与客户之间缺乏沟通环节。

瀑布模型也演变出一种新的变种：V-Model。支撑V-Model的背后的“启发“主要是每进行一步就开始进行将来的测试的计划、设计。比如说：在软件的实施阶段，就在开始计划单元测试。

增量模型 Incremental Model

本质上是迭代的

从第一个”增量“开始，我们给客户交付的东西就是一个能使用的软件了。但一开始给客户看的嘛，时间不足所以只能先实现核心的功能，在后续发布的增量，就是一个一个客户需要完成的新功能的实现。

优点：

时间要求较高的时候可以使用的一个过程模型。

失败的可能性较低。

困难：

在交付新增量的时候，要考虑好如何在不破坏原有增量的前提下这么做。

无法处理变化的需求。

RAD模型

是Linear Sequential的高速变种。主要思想是通过构件实现多团队并行开发，速度是真的快。

缺点：

不适用于技术风险高的情况

有的时候系统无法模块化

性能一般

快速原型法 Prototyping

本质上是迭代的。

如果用户对于这个产品只有模糊的认识，可以用这样的方法：快速搭建一个原型产品，例如说快速建立一个能用的用户界面，里面功能、性能可以先粗略地搞定，将原型交付、和客户进行进一步沟通。

优点： 这个方法可以精炼出用户的需求，由于这个模型的一些问题，我们往往只将它作为一个导出需求的方法。同时，它也是用户和设计之间交换最频繁的模型。

缺点： 性能差。

敏捷开发

需求分析

综述

需求分析需要完成以下几个任务：
1.Inception：初启。

2.Elicitation：导出。从共利益者那里得到需求，可以采用的方式有：QFD（quality function deployment），CRG（collaborative Requirement Gathering），面谈，观察和问卷等形式。

3.Elaboration：细化。要表示出需求的信息、数据关系，功能和行为。

4.Negotiate：谈判。谈判相信大家都熟悉，但和客户的谈判切记，不要“威慑”客户，记住我们要的是达成一致，共赢。

5.Specify：具体化。简单来讲，就是将需求写进一个文档中，正式地进行保存。

6.Validation：验证。比较重要的一点就是确认自己理解的需求和客户的需求是否一致。

除此之外，还有第七个单独拎出来的：

Requirement Management：需求管理。主要是管理变化的需求，具体一些就是需要导出、编档、组织、跟踪变化的需求。

需求的分类、分级

需求的分级：

首先是 共利益者的需要： 客户提出的无关于解决方法的

系统的特点(Features)：软件的实现方案，也是相关功能的集合

软件的需求： 客户对于软件的显式和隐式需求都要无歧义的表达出来。

需求的分类：

功能需求：描述的就是软件作为一个数据的“处理器”，将一个输入转换成软件开发人员想要的输出。功能需求就是讲的输入-输出。

非功能需求：描述的是质量和属性的需求。我的理解是非功能需求要大于功能需求。举个例子，就是这里的需求可以是“识别速度达到100fps”这样的。

设计约束：限制了在构建时进行的选择，比如说我就要你用sqlite这个数据库来开发。

需求分析模型

我们想要表述一个需求，因此提出了这么一个叫分析模型 的东西，它的任务就是给我们任意时刻的需求的一个“snapshot”。这是分析模型作为一个整体。

为了全面的表达一个需求，我们需要从几个方面进行描述。这些方面不一定要全部运用，这样的好处很明显——就是能够明确需求。这些“方面”正规的表述是分析模型的 “元素”，这样应该更好理解一些。

基于场景的元素 Scenario-Based Elements

场景就是一个用户在使用系统的时候的一个“故事“，可以理解成系统是如何被使用的。

这样的元素常用的就是用例 Use Case。用例是一个外界的使用软件的人（或其他可能的事物）如何与软件进行交互。这样的人或物被称为actor。UML中，经常画用例图进行描述。

同时，系统所实现的功能也属于这一类元素。Functional：表达了软件的功能是如何被处理的。UML中，经常画数据流图、活动图来表示。

基于类的元素 Class-Based Elements

能够提取出一些共同的属性、行为的东西被称为一个类。比如说人可以是一个类（Class），人这个类有一些共有的属性，比如都有身高、体重、出生日期等等。

在需要从一段文字的表述中提取出类来，需要识别的是名词。

而识别出类的方法需要识别的是动词。可以借助UML的时序图、协作图来分析。

在UML中用类图表示。

说到类，可以提到的是一个叫CRC卡片的东西，它常用来分析一个类的职责和他的协作者。CRC Class Responsibility Collaborator 卡片分为三个部分：类名、类的职责和类的协作者。下面细说：

类的职责： 这个类所需要完成的任务，比如需要存储的信息（属性）和信息如何被利用。比如一个星际争霸里的狂热者，他拥有攻击力、护盾、生命值的属性，能做的是冲锋、用灵能利刃攻击，这些都要写进类的职责里。

类的协作者： 指代另一个类，我们手头的这个类需要从协作者那里得到信息。比如说：我的整个部队需要保存所有狂热者的信息，狂热者就是这个部队类的一个协作者。

行为元素 Behavioral Elements

先介绍一下”状态”在这个情景之下的定义。状态是外部被观察到的行动模式，举个例子，就是比如状态图可以有一个“系统关闭状态”，这个状态可以由一个其他状态经过“关机”操作获得。

在行为元素中，我们常用UML的状态图State diagram来进行表示。表示的有状态，还有状态之间切换的触发事件、状态切换时的行动等。

流元素 Flow-oriented Elements

这样的元素讲的主要就是信息是如何变化、处理的。比如说一个输入x，经过了怎样的变换得到输出，在这里可以就是x+1。

常用的UML图有：DFD图（Data Flow Diagram）

常用的分析方法：SA和OOA

分别是结构化分析方法（Structured Analysis）和面向对象分析方法（Object Oriented Analysis）。

设计相关

设计就是把一个即将被建造出来的东西表示出来。在软件工程中，设计主要偏重于四个主要方面：数据、架构、接口、组件 。

设计与之前所学的需求分析存在关系：客户需求和软件的设计存在着紧密的联系。元素之间的映射如图所示，比如说基于场景的需求元素就会影响到用户界面的设计

先简单介绍下这几个主要方面的作用吧：

数据设计 设计数据结构，在软件中实现。

架构设计 描述了需要有什么组件，这些组件之间是什么个关系。

接口设计 描述的是程序的组件之间如何协作、交流。从一个更像“程序”的角度来讲，就好像是一个类，别的类需要访问它里面的一个值，就可以是在这个类里写一个公有函数，返回值就是需要访问的值。还描述了软件如何与使用的人进行交互，也就是我们所熟知的UI（User Interface）。

组件设计 组件就是将所要执行的步骤封装起来，设计软件的人就不用太注意每个组件内部的实现，只需要考虑接口，

设计的过程：Outline

软件设计是一个迭代的过程，其中，需求在构造软件的时候被转化为设计的“蓝图”。比如说一开始，系统的抽象层次还很高。一步步迭代下去，后面的就会不断细化。但要注意的是，需求是贯穿了整个设计的，越往后，设计和需求的联系会逐渐降低。

软件质量

这里我们引入软件质量的概念，因为在设计的过程中，我么会经常评估“这个设计的质量如何”，并引入一些技术性的评论。一个好的设计要做到以下三点：

实现所有明确说出来的需求

应该对于程序员来说是可读的、可理解的，它来引导程序员来编程、测试和支持这个软件。

需要提供一个这个软件的全貌（complete picture）

以上三条可以理解为设计过程的目标。

设计的“行话”和概念

抽象

抽象的意思我们都比较了解，抽象程度越低，就代表我们在提供越详细的解决方案。一个逐渐降低的抽象程度的例子是：用户故事->用例->伪代码。

在设计过程中，抽象也分两种：过程抽象和数据抽象，过程抽象只会用一个动词，比如使用相机，但具体用相机干什么，怎么用，这里就进行了抽象。而数据抽象是一系列的数据，这些数据用来描述相机这个东西，而上面说的使用相机这个过程，使用这一动作需要相机相关的属性实现。比如需要相机的位置

模式 Pattern

设计模式描述了解决某个特定环境中的特定设计问题的设计结构。

每一个模式都应该提供一个简介，以帮助设计人员决定：
我目前的问题是否适合用这个模式解决

这个模式是否能被重用

这个样式是否能作为一个指南，来开发一个相似的系统

关注点分离 Separation of Concern

一个问题如果太大了，就不好解决，但分成几个小问题就好办多了，分成小问题，之后再分别解决。一个关注点是一个软件的行为或特点，在需求分析中被我们提到。

模块化 Modularity

一个软件被模块化了，才能智能的管理它。如果一个软件由一个单一的模块组成，那么它将无法被软件工程师轻易的掌握。

随着软件模块数增加，集成与组织这些部件的成本会越来越高，而开发每个组件的成本与难度都会降低。所以为了最低的支出，折衷吧。

模块化的优点：开发的计划可以更容易、增量发布更容易、单元测试和维护更容易，变化也能更好的适应。

信息隐藏 Information Hiding

在我们将一个软件设计进行模块化之后，一个问题是：我们如何设计，才能得到最好的一组模块？信息隐藏这个原则就出现了。它的意思是，其他的模块不能访问这个模块的信息（算法和数据），如果说其他的模块不需要用这些信息的话。

为什么要信息隐藏？当我们对软件进行更改的时候，因为大部分不需要被访问的数据、算法都没有被访问，一个被引入的错误将不会被传播到软件的其他部分。就是减少了测试和维护的成本。

功能独立性 Functional Independence

功能独立性主要就是，其他模块能够访问某模块的时候，只能看到一个接口，每个模块设计的时候只要考虑自己需要完成的一部分功能就好。这里引入功能独立性的两个评判标准，也是模块设计最好能做到的：高内聚、低耦合

内聚： 是指模块内部是否专心完成自己需要完成的部分。内聚越高，代表模块越专心完成自己的部分。这样一来，内聚越高，就代表功能越独立。

耦合： 是指模块之间进行合作的程度，就是相互依赖的程度。耦合程度越低，代表功能越独立。耦合程度是直接关系着功能独立性的。内聚只是次要影响因素。

求精 Refactoring

是一种自顶向下(top-down)的设计策略。就是一个细化的过程。

重构 Refactoring

重构是一种重新组织的技术，意图是要提高一个代码的内部结构，而不改变它的行为。
一种观点：好系统（代码）是重构出来的

架构设计

软件架构

一个软件或者程序的架构就是这个系统的结构。它包含了组件元素、组件元素之间的联系和元素外部可见的属性。

架构的”风格“：介绍

架构的风格类似于一种分类，每一种架构风格都会有自己的：

组件集合 Set of components 组件就是系统中，完成一个功能所需要的一个模组（比如说：数据库）。

连接器的集合 Set of Connectors 这些东西允许组件之间的交流。

约束 Constraints 定义了组件们如何合成一个系统

语义模型 Semantic Models 允许一个设计者能够理解这个系统的整体属性。

架构风格的分类：

以数据为中心：Data-Centered

中间有一个数据库，其他需要更改信息的组件频繁地访问它。

数据流：Data Flow

输入数据经过了一系列的用来计算的元件，形成一个输出。

调用-返回值：Call-Return

一个主程序会调用许多子程序，也就是我们编程的时候总会弄的“routines”或者函数。这样的一个长得像一个树的结构就形成了一个调用-被调用的结构。函数A的执行过程中，传入参数调用函数B，函数B再调用C，类似这样的过程。

面向对象：Object-Oriented

系统的组件会封装数据和操作（实际上我们来回想一下类，类不就是有类的属性和方法嘛），组件之间靠传递消息来沟通。

分层架构：Layered Architecture

每一个层都完成一系列指令，越往下走越接近机器指令。具体来讲，就是外层执行的是用户界面的指令，里层，执行的就是和OS的界面，比如调用OS的一些函数。

组件设计 Component Level Design

什么是组件

从不同的角度看构件，解决“什么是组件”这个问题。组件是系统里一个模块化的、可部署的、可被更换的零件，内部的实现无法被看见，被封装了，和外界之间用接口联系。

从面向对象的角度来讲，组件就是一系列互相协助的类。我们进行组件的设计就是对需求分析的分析类进行精化，需要更多的分析步骤来表示出被精化的设计类。

具体步骤是，由分析类知道，一个类需要实现的方法和具有的属性，之后进行精化，得出组件的接口，之后精华设计类，把较为具体的方法名称列出，但当然了，目前不需要具体实现出来这些方法，但需要细致到把组件的接口更具体的写出来，要实现什么函数才能实现这个接口。（可以类比下java里的interface）

从传统的观点来讲，组件就是一个实现：处理逻辑、内部数据结构、接口的一个元素。

从进程相关的观点来讲，软件就是用以后的组件做成的，组件有重用的能力。它有接口、函数、和其他组件交流。

软件测试

软件错误会造成严重的后果，而测试，从这个名字就能听出，就是在交付给用户前，为了找到错误并修正的一个活动。测试软件的人不一定是开发者自己，也可以是独立的测试人员。开发者测试的时候，可能只会“轻轻地”进行测试，而独立的测试者，在测试之前肯定得先学习关于这个系统的结构啥的，尝试挑错。

测试：基本

两个概念：验证和确认

测试的Verification 和 Validation，这两者之间存在差别。Verification是验证的意思，验证的是一个软件是否能够正确地完成一个指定的功能，而Validation是确认，在前面的需求分析阶段我们了解了客户的需求，现在需要验证我们的设计是否满足了需求。另一个简单的区分方法是把这两个行为变成两个问句：“我们是否在正确地实现产品” 和 “我们做出的产品是不是客户要的”。

测试的步骤

步骤大体如下：先进行单元测试，测试好的单元根据设计得到的结构信息进行组装、集成，进行集成测试。集成测试之后得到一个装配好的软件，对照着软件需求进行确认测试，确认的软件进行系统测试，最终得到了可运行的系统。

如何知道一个软件测试完毕

关于这个问题，有三个说法：一个是你永远测试不完一个软件，你交付了软件，用户就是在为你进行测试。说法二是：你时间用完了，或者是你没钱了，你才算做完了测试。三是：用统计模型，我有95%的自信来说我测完了。

单元测试 Unit Testing

传统单元测试

一个单元，它的测试一般无法独立完成，需要上面有驱动程序(Driver)来传入参数、调用这个组件，下面有桩(Stub)来模拟这个组件的下层的结构。

面向对象单元测试

如果你有面向对象编程的经历，那么就会知道在面向对象编程的时候，一个肯定会出现的概念是类，类里面写了函数，这些函数用类的变量和类里的其它函数一同完成一个指定的功能。

所以在测试一个面向对象的程序的单元的时候，必须结合它所在的类进行测试。举这样一个例子：X是一个父类里的方法，子类很多，都继承了父类。这些子类在执行同一个方法X的时候，可能X里面需要的变量、甚至X自己都可能被重写了，所以必须结合类进行测试。

实际上，OO的时候就不说”单元测试“了，更多是直接说类测试。类测试是被类里面封装的操作和类本身的状态、行为驱动的。这一点是和传统的单元测试不同的，传统的单元测试更关注一个模组的算法实现和数据流。

集成测试 Integration Testing

自顶向下方法：Top-down Approach

首先测试顶部模块，下面还没有测试的模块用存根(Stub)来“模拟”底层的功能。测试完成之后需要用实际的还没测试过的模块来代替存根，这里的代替可以是深度优先，即先一路向更底层的模块替换，也可以是广度优先，即先将根部的模块的子模块替换上来。

自底向上方法：Bottom-Up

就是先把底层模块合成簇：Cluster，每一个簇实现一个子功能，用一个驱动程序(Driver)来调用子功能簇。测试完这个之后，去掉驱动程序，组合这些子功能簇形成一个巨大的子功能簇。

三明治测试

测试的策略：白盒与黑盒，动态与静态

静态测试与动态测试

静态测试的方法可以包括：代码审查、代码走查、技术评审、桌面检查。
动态测试包括白盒、黑盒测试。

白盒测试：已知软件结构的测试

白盒测试是一种设计测试用例的方法，它需要借助我们所知道的控制结构来设计出测试用例。一般来讲，软件的白盒测试可以做到的：

1.保证每一个组件的每一个独立的路径都被执行一次。这里独立路径的概念，在后面会介绍，现在可以简单地认为：一条路径要想独立，那么它就得经过至少一个别的路径没有的节点。

2.执行每一个逻辑的分歧点，就是一个条件分支无论是否满足都会被测试一遍。

3.执行每一个循环的边界和循环的边界内部。

4.执行内部的数据结构，验证它们可用。

下面介绍几个白盒测试的方法：一个最基本的测试是：Basis Path Testing

基础路线测试

每一个独立的路径都被执行一次，执行出来的效果其实就是：每一条语句至少都过一次。

具体的方法是：

1.首先画出一个算法/组件的流程图。

2.之后确定这个算法的秩复杂度，这一步的目标是，一个组件的独立路径条数不可能大于秩复杂度。秩复杂度的计算有两个简单的方法：一个是计算带有判断条件的节点的个数+1，这个值就是秩复杂度，另一个方法是数出流程图的线所圈出的区域的个数。

3.知道秩复杂度之后，确定一系列无关的路线。这个无关路径不唯一，因为最终的目标其实就是把每一行代码都跑一遍，比如说有A、B、C、D四个组件需要测试，你测试的第一条路径是A->B,第二条路径是A->B->C，第三条路径是A->B->D，和你测A->B,A->B->C->D的效果一样。

4.最后是根据刚刚确定的几条路线，设计输入值来强迫程序走这几条路线，从而完成测试。

条件分支测试

测试一个程序模组里的逻辑。

数据流测试

依照变量定义、变量值的改变的位置设计测试路径。

循环测试

循环测试是一个用来测试循环体结构是不是可用的，把循环分成两种：简单循环和嵌套循环。

简单循环的测试可以用以下的几种测试：跳过循环 、 只执行一次循环 、 执行两次 、 执行m次（m小于n,n为循环最多时候的执行次数） 、 执行n-1 、 n 、 n+1次。嵌套循环的测试可以不像简单循环那样一个一个的执行，可以做的是，从最里层的循环开始，先把其它的循环置为最小值，之后再一个一个循环往外执行。

黑盒测试：未知软件结构

黑盒测试专注于测试软件的功能需求有没有被完成。与白盒测试相反的是，黑盒测试刻意地无视控制结构。

黑盒测试的关注点放在了：有没有执行错误的或者被遗漏的功能？有没有组件之间的接口的问题？有没有数据库或者数据库访问的错误？行为或着性能的问题？初始化或者是终止的问题？

接口测试

一个传统的函数有输入的参数和返回值，接口的测试就是检查一个程序组件能否以正确的顺序接受传递进来的信息，并且以正确的顺序和数据格式返回值

等价类划分

黑盒测试不知道软件的结构，就无法根据控制的结构来进行测试用例的设计。那么用例的设计就只能基于输入数据进行划分，等价类划分就是将一个输入域划分成几个情况。一个理想的测试用例可以单独检测出一类错误。

那么怎么进行等价类划分呢？在分类时可以通过Vaild和Invalid进行分类。测试的时候，根据可用的数据的域（一个值，一个值域……）决定是应该测试一个可用的、两个不可用的值还是一个可用的输入、一个不可用的输入。

二者之间的关系、对比

从测试的阶段考虑：黑盒测试适用于各测试阶段，白盒测试通常只能用于单元测试。

从性质上对比：黑盒测试测试的是“我们是否在搭建一个正确的系统”，是一个验证技术，白盒测试测的是“我们是否在正确地搭建系统”，是一个确认技术。
